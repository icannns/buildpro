1. PEMBUKAAN (30 detik)
================================================================================

"Selamat pagi/siang, Pak/Bu Dosen dan teman-teman sekalian.

Hari ini saya akan mempresentasikan BuildPro, sebuah sistem manajemen 
konstruksi dan renovasi yang kami kembangkan menggunakan arsitektur 
microservices.

Yang membuat BuildPro unik adalah, kami mengimplementasikan 4 bahasa 
pemrograman berbeda dalam satu ekosistem: Node.js, Java, Python, dan Go. 
Ini mendemonstrasikan konsep polyglot microservices dan bagaimana berbagai 
teknologi bisa bekerja sama dalam satu sistem terintegrasi."


================================================================================
2. ARSITEKTUR SISTEM (1 menit)
================================================================================

[Buka README, scroll ke diagram arsitektur]

"Mari kita lihat arsitektur BuildPro. Sistem ini terdiri dari 9 komponen 
utama yang semuanya berjalan dalam container Docker:

Pertama, di lapisan frontend, kita punya React Client yang dibangun dengan 
Vite untuk performa optimal.

Kedua, semua request masuk melalui API Gateway yang dibuat dengan Node.js. 
Gateway ini bertugas routing request ke service yang tepat.

Ketiga, kita punya 5 microservices dengan bahasa berbeda:
- Auth Service menggunakan Node.js untuk autentikasi JWT
- Project Service, juga Node.js, mengelola timeline proyek dan log harian
- Budget Service menggunakan Java untuk manajemen keuangan dan termin pembayaran
- Material Service berbasis Python Flask untuk inventory management
- Dan Vendor Service menggunakan Go untuk portal supplier

Keempat, ada GraphQL Server untuk query layer yang lebih fleksibel.

Dan terakhir, semua service terhubung ke MySQL Database sebagai single 
source of truth.

Semua ini diorkestrasi oleh Docker Compose sehingga deployment sangat mudah."

PERSIAPAN Q&A
================================================================================

PERTANYAAN: Kenapa pakai 4 bahasa berbeda? Apa tidak lebih mudah pakai 1 bahasa?

JAWABAN:
"Pertanyaan bagus. Ada beberapa alasan:
Pertama, untuk mendemonstrasikan polyglot architecture yang sering dipakai 
di perusahaan besar. Di dunia nyata, sistem enterprise sering merupakan 
kombinasi dari berbagai teknologi legacy dan modern.

Kedua, setiap bahasa punya kelebihannya masing-masing. Java sangat baik 
untuk financial systems karena presisi dan stabilitas. Python excellent 
untuk data processing. Go bagus untuk high-performance services. Node.js 
ideal untuk I/O intensive operations.

Ketiga, ini memaksa kami untuk benar-benar memahami service boundaries 
dan API contracts, karena services harus berkomunikasi via REST API 
terlepas dari bahasa apa yang mereka pakai."

---

PERTANYAAN: Bagaimana services berkomunikasi satu sama lain?

JAWABAN:
"Services berkomunikasi via REST API melalui API Gateway. API Gateway 
bertugas routing request ke service yang tepat.

Semua services juga share satu MySQL database sebagai single source of 
truth, jadi mereka bisa read/write data yang sama dengan proper foreign 
keys untuk maintain data integrity.

Untuk complex queries, kita juga punya GraphQL layer yang bisa aggregate 
data dari berbagai tabel dalam satu query."

---

PERTANYAAN: Kalau satu service down, apa yang terjadi?

JAWABAN:
"Karena arsitektur loosely coupled, kalau satu service down, service lain 
tetap bisa berjalan untuk fungsinya masing-masing.

Misalnya kalau Vendor Service down, Project Service dan Budget Service 
tetap bisa beroperasi normal. User hanya tidak bisa akses vendor portal 
sementara.

Untuk production environment, kita bisa implement health checks dan 
auto-restart mechanism di Docker atau menggunakan orchestration tool 
seperti Kubernetes untuk high availability."

---

PERTANYAAN: Apakah sistem ini bisa di-scale untuk handle lebih banyak users?

JAWABAN:
"Tentu saja. Karena semua services sudah containerized, kita bisa easily 
scale individual services.

Misalnya kalau Material Service banyak diakses, kita bisa spin up multiple 
instances dari Material Service saja tanpa perlu scale yang lain.

Dengan menggunakan Docker Swarm atau Kubernetes, kita bisa implement 
horizontal scaling dan load balancing dengan mudah.

Database juga bisa di-scale dengan read replicas atau sharding jika 
diperlukan."

---

PERTANYAAN: Bagaimana handle data consistency?

JAWABAN:
"Kita menggunakan shared database dengan proper foreign keys dan 
constraints untuk maintain data integrity.

Untuk operasi yang membutuhkan consistency, kita bisa implement database 
transactions.

Dan kita sudah menyiapkan event_queue table untuk implement eventual 
consistency pattern jika needed di future development."

---

PERTANYAAN: Berapa lama development BuildPro ini?

JAWABAN:
"Development BuildPro memakan waktu [sesuaikan dengan waktu actual Anda], 
dengan fokus pada implementing semua requirement yang diminta plus beberapa 
bonus features seperti real-time updates dan GraphQL.

Yang paling challenging adalah mengintegrasikan 4 bahasa berbeda dan 
memastikan EAI bekerja dengan smooth."

---

PERTANYAAN: Apa kendala terbesar yang dihadapi?

JAWABAN:
"Kendala terbesar ada dua:

Pertama, ensuring consistent API contracts antar services. Karena pakai 
banyak bahasa, kita harus extra careful dengan data types dan API responses.

Kedua, implementing real-time EAI integration. Kita harus design bagaimana 
Budget Service bisa detect milestone achievement tanpa constant polling, 
dan ensure payment status updates reflected immediately di frontend.

Tapi dengan proper database design dan event-driven thinking, masalah-masalah 
ini bisa diatasi."
